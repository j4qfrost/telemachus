# Hardening checklist — telemachus (self-brands "Odysseus")

Language: Python 3.12 (CI) — venv on dev box is 3.14
Framework: FastAPI / Starlette on uvicorn, SQLAlchemy, served as a long-running daemon
Fork of: `pewdiepie-archdaemon/odysseus` — harden the fork deltas (on `main` vs `upstream/master`) carefully; do not rewrite upstream behavior.
Workload shape (per `lang-assessment-2026.md`): **"long-running production daemon, handles untrusted input."** This is the single riskiest archetype for unhardened Python — the assessment explicitly says "avoid unhardened Python here." That ranking drives the priorities below: supply chain and input-path robustness come first.

Generated by the production-hardener pass on 2026-06-25. Checked items were verified by running the project's own checks on this commit.

---

## Priority (fix these first)

1. **[HIGH] Bump the 5 vulnerable runtime deps — 8 known CVEs, two on the untrusted-input path.**
   `pip-audit -r requirements.txt` flags `starlette` (CVE-2026-54283 / -54282), `python-multipart` (CVE-2026-53540), `cryptography` (GHSA-537c-gmf6-5ccf), `pydantic-settings` (GHSA-4xgf-cpjx-pc3j), and `pypdf` (CVE-2026-54531 / -54530 / GHSA-jm82-fx9c-mx94). `starlette` and `python-multipart` parse *every inbound HTTP request / multipart upload* — exactly the untrusted-input surface the assessment warns about. **Not hand-edited**: `requirements.txt` is pip-compiled and there is no `pip-compile` on this box; blind line edits would desync the lock from `requirements.in` and the dev lock's `-c requirements.txt` cross-refs. Fix is a re-resolve, not a patch (command below).

2. **[HIGH] No supply-chain scan in CI — that's why the 8 CVEs sat undetected.** The lockfile is fully pinned (good) but never scanned, so CVEs published after the last compile are invisible. *Partially fixed in this pass:* an advisory `audit` step (`pip-audit`) was added to `.woodpecker/ci.yml`. Flip it to a hard gate (drop `|| true`) once priority #1 lands.

3. **[MED] No coverage measurement / gate.** 329 tests pass and the suite is genuinely strong on auth/security/owner-isolation regressions, but there is no `--cov` run and no floor. On an untrusted-input daemon you want a measured number and a ratchet so the error-path coverage below can't silently regress.

4. **[MED] 249 silent `except: pass` / `except: ...` blocks across the input path.** Most are legitimate best-effort cleanup (e.g. `OSError` on `os.path.getsize`, session-cache clears) and the one in the auth middleware is correctly *fail-closed*. But a 249-count blanket means a real bug on a hot path can be swallowed invisibly. Needs a triage pass (debug-log the swallowed exception where the intent is "best effort", keep `pass` only where genuinely unreachable-or-irrelevant). Judgment-heavy — left as a checklist item, not auto-edited.

5. **[LOW] Dev/CI Python is 3.12 but the dev venv resolved on 3.14.** Two `datetime.utcnow()` DeprecationWarnings already surface (`src/task_scheduler.py:238`, SQLAlchemy default). Pin the dev interpreter to match CI (3.12) to avoid resolver/behavior drift, and migrate the deprecated `utcnow()` calls before 3.14 makes them hard errors.

## Static analysis
- [x] Lint passes — `ruff check .` → **All checks passed!** (on this commit).
- [ ] **Lint scope is intentionally minimal (`select = ["F"]` only).** `pyproject.toml` documents the deferral: ~4k E501, ~440 import-sort, the F401/F841 sweep are all out of scope for the CI bootstrap. Expand `select` (at least `E`, `I`, `B` for bugbear) in a dedicated format PR — bugbear (`B`) in particular would flag risky patterns on the input path.
- [ ] **No format check.** `ruff format --check` is not run anywhere. Add it (after the format PR lands) so style can't drift.
- [ ] **No type check.** No `mypy`/`pyright`, no type-check CI step. Per `lang-assessment-2026.md` § AI alignment + mental-model #2, strict typing is the single biggest lever for Python robustness ("runtime errors that typed languages catch statically" is Python's named weakness). A full strict pass on 79k LOC is a project; start by type-checking `core/` and `routes/auth_routes.py` (the security core) under `pyright`.

## Deploy
- [x] **Loopback bind is load-bearing and correctly defaulted** — the host has a PUBLIC eth0 with no firewall, so this matters. systemd unit binds `127.0.0.1` with an explicit "never 0.0.0.0 on a privileged box" comment; `install-service.sh` defaults `BIND_HOST=127.0.0.1` and refuses to install a unit with unresolved tokens; docker-compose publishes `${APP_BIND:-127.0.0.1}` (container itself binds 0.0.0.0, exposure controlled at the publish layer — correct). All sidecars (chromadb, searxng, ntfy) publish to loopback.
- [x] Container drops privileges — `docker/entrypoint.sh` drops to PUID/PGID via gosu; `exec uvicorn` keeps it PID-1-child so SIGTERM reaches it.
- [x] Reproducible-ish build — base images `@sha256` pinned; deps pinned. (Strengthen with `--require-hashes`, above.)
- [x] systemd unit hardened-adjacent — `Restart=always`, `EnvironmentFile=-`, optional tmpfs secret-render path (sigil/Vaultwarden) documented.
- [ ] **Add systemd sandboxing directives.** The unit has none of the cheap wins: `NoNewPrivileges=true`, `ProtectSystem=strict`, `ProtectHome=read-only` (with `ReadWritePaths=` for `data/`+`logs/`), `PrivateTmp=true`, `RestrictAddressFamilies=AF_INET AF_INET6 AF_UNIX`. For a privileged daemon on a public-IP host these are high-value, low-risk. Judgment needed on `ReadWritePaths` (Cookbook writes under `data/.local`, `.ssh`, HF cache) — flag, don't auto-apply.
- [ ] **Artifact signing** — no Sigstore/cosign on the released OCI image. Add `cosign sign` (keyless/OIDC) in `release.yml` for the container.
- [x] Release process documented — referenced in `docs/RELEASING.md` and inline in `release.yml`.
- [ ] **Rollback path not documented** — releases are tagged + image-pushed, but there's no documented "pin to previous image tag / redeploy" procedure. Add a short rollback section to `docs/RELEASING.md`.

## Observability
- [x] Logging framework present — stdlib `logging` (`getLogger`) used across most of `routes/` and `app.py`; `logs/` dir created on launch and gitignored.
- [ ] **Logging is inconsistent** — some hot paths use `print()` (see Error paths) and 249 swallowed exceptions log nothing. Standardize on `logging`, and ensure swallowed exceptions log at `debug`.
- [ ] **No structured (JSON) logging.** stdlib text logging is fine for single-host, but for an unattended daemon consider a JSON formatter so logs are queryable. Low priority for self-host.
- [ ] **No health-check endpoint verified.** Daemon needs a `/healthz` (or equivalent) for systemd/compose liveness; compose healthchecks exist for sidecars (searxng) but not for telemachus itself. Confirm/add a cheap liveness route and wire a compose `healthcheck`.
- [ ] **No metrics endpoint/hook** — no Prometheus/OTLP. Acceptable for v0.1 self-host; revisit if it ever runs multi-user/unattended at scale.
